How to Diagnose a Process Before Automating It

6 min read · Process Improvement · BESSÓ Systems
VSM PDCA Process Mapping TIMWOOD Lean

The common mistake: automate first, understand later

The most frequent failure pattern in industrial digitalization is not a technology failure — it is a process failure. Teams select a platform, budget the project and begin integration before anyone has properly mapped what the process actually does today. The result is an automated version of a broken process, delivered on time and on budget.

Automation does not fix poor flow, missing standards or unclear accountability. It locks them in. A good process diagnosis, done before any technology decision, is the highest-leverage activity in a transformation project.

Step 1 — Map the current state

Go to the floor. Do not rely on the existing documentation — most process documents describe the intended process, not the actual one. Use a Value Stream Map (VSM) to follow the material and information flow from raw input to finished output.

At minimum, record:

  • Each operation step and its operator
  • Observed cycle time (not standard time)
  • Wait times and queue lengths between steps
  • Where the process deviates from the documented version
  • Information triggers: what tells each step to start?

If the process is too long to map in one observation session, split it. Do not fill gaps with assumptions.

Step 2 — Identify waste

Use the TIMWOOD framework as a structured lens: Transportation, Inventory, Motion, Waiting, Overproduction, Overprocessing, Defects, and — often the most costly — non-Utilised talent.

For each waste type identified, note the location, frequency and estimated impact. Do not attempt to solve anything yet. The goal at this stage is to see clearly.

Step 3 — Measure what matters

Three baseline measurements are almost always worth collecting before proceeding:

  • Takt time: the available production time divided by customer demand. This is your target rhythm.
  • Cycle time at each step: how long each operation actually takes. Compare it to takt. Where cycle time exceeds takt, you have a bottleneck.
  • Defect or rework rate: what fraction of units require correction at each stage.

These three numbers, even roughly estimated, will define most of the diagnostic conversation that follows.

Step 4 — Root cause analysis

For the problems identified, resist the first plausible explanation. Use 5-Why analysis to follow the chain of causes to a root level, or a fishbone diagram when multiple causal categories (machine, method, material, human, environment) are in play.

Root cause analysis is most useful when done with the people who operate the process — not as an interview subject, but as co-analysts. They will surface causes you will not find from observation alone.

Step 5 — Define the target state

With the current state understood, you can specify a target state: what the process should look like once the identified problems are resolved. This is distinct from designing an ideal state. A target state is achievable in the near term with the existing constraints. An ideal state is useful for direction, but not for planning.

Document the gap between current and target with specific, measurable terms. Avoid vague improvement language ("more efficient", "less waste"). Specify: what operation, what metric, from what value to what value.

"Automation amplifies both strengths and weaknesses. A well-understood process automated well performs predictably. A poorly-understood process automated fast creates problems at machine speed."

Only then: design for automation

Once the target state is defined, the technology question becomes much more tractable. Instead of asking "what system should we buy?", you can ask "what does this specific target state require in terms of data, logic and control?" The answer is usually much simpler — and much cheaper — than the platform selected at the beginning of the project.

Process diagnosis is not a preliminary phase before the real work. It is the real work. Everything else is implementation.
